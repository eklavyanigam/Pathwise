1. Abstract

Pathwise is a full-stack web application designed to help students and early-career learners understand how ready they are for specific technology roles. The application solves a practical problem faced by many learners: people often know which role they are interested in, but they do not know whether their existing skills are enough, which gaps matter most, or what they should learn next in a structured way.

The system accepts a target role and a list of user-entered skills, compares those skills against role-specific expectations, calculates a readiness score, identifies matched and missing competencies, and generates a practical action plan. The action plan includes a final summary, category breakdown, missing-skill prioritization, “what if I learned” simulation, role insights, skill learning roadmap, learning resources, and suggested portfolio projects.

From a technical perspective, Pathwise combines a static frontend with backend-style capabilities through Supabase. Authentication is handled using Google OAuth and email/password methods, with support for guest mode. Signed-in users have persistent progress saved to the cloud, while guests store progress locally in the browser. If a guest later signs in, Pathwise merges guest progress into the user account.

The implemented version of Pathwise includes a dedicated intro page, a three-step analyzer flow, a profile page, progress persistence, session restoration, print-friendly architecture, and a focused user experience built with HTML, CSS, JavaScript, Supabase, and Vercel.

2. Problem Statement and Need

Students frequently struggle to answer four practical questions when planning for careers in technology:

  1. Which role should I target?
  2. How ready am I for that role right now?
  3. Which skills matter most for that role?
  4. What should I learn next in the right order?

Many platforms provide general career advice, articles, or course catalogs, but very few translate a student’s current skills into a concrete, role-specific readiness assessment. This creates confusion, wasted effort, and unstructured learning.

Pathwise addresses this gap by combining role-based skill mapping, scoring, prioritization, resource suggestion, and progress persistence in one guided workflow. Instead of only showing information, Pathwise aims to support a decision: where the learner stands and how to move forward.

3. Project Objectives

  • Provide a clean and guided user experience for career readiness evaluation.
  • Allow users to choose a target role from a curated list of technology careers.
  • Collect current user skills and proficiency levels using a structured input flow.
  • Compute a readiness score using weighted role requirements and user skill levels.
  • Highlight missing skills and prioritize them according to role importance.
  • Generate an action plan showing what to learn, where to learn it, and what to build.
  • Support both authenticated users and guest users without breaking the experience.
  • Persist progress across reloads and restore the user’s last state.
  • Use secure cloud persistence with row-level access control.
  • Present the output in a way that is useful for students, internship applicants, and early-career candidates.

9. Architecture Overview

Pathwise follows a client-heavy web architecture. The frontend owns the user interface, state transitions, scoring presentation, and direct Supabase client integration. Supabase acts as the managed backend service for authentication and storage.

USER | v Browser (HTML + CSS + JavaScript) | +--> Intro / Auth Layer | | | +--> Google OAuth via Supabase | +--> Email auth via Supabase | +--> Guest local session | +--> Analyzer Layer | | | +--> Setup | +--> Analysis | +--> Action Plan | +--> Profile | +--> Persistence Layer | +--> localStorage (guest) +--> Supabase Auth + Database (signed-in users)

Why this architecture was chosen

  • Fast MVP implementation without maintaining a custom server.
  • Supabase provides reliable auth and storage primitives out of the box.
  • The analyzer logic is mostly deterministic and can safely run on the client.
  • Direct database access is safe because Row Level Security constrains access per user.

11. Authentication Design

Pathwise supports three access paths:

  • Google sign-in
  • Email sign-in / account creation
  • Continue as Guest

Google authentication

Google authentication is implemented through Supabase OAuth. On click, the browser is redirected to Google through Supabase. After successful authentication, the application restores the session and transitions the user from the intro page into the analyzer.

Email authentication

Email sign-in and signup are implemented through direct Supabase auth endpoints. The system supports signing in with an existing email and creating a new account using the same UI. Password reset is also supported via Supabase email flow.

Guest access

Guest mode is a key usability feature. It removes friction for first-time users and allows the analyzer to be used instantly. Guest progress is stored locally and may later be merged into a signed-in account.

Session behavior

  • Sessions are persisted with Supabase client auth persistence.
  • Guest mode sets a local resume mode.
  • Reload attempts to restore the correct analyzer page and state.
  • Top notices communicate auth status without browser alerts.

13. Guest Storage and Data Merge Logic

Supporting guest mode required a second persistence strategy in addition to cloud save. Pathwise uses browser localStorage for guests and Supabase for signed-in users.

Guest storage keys

  • pw_guest_progress — serialized guest progress payload
  • pw_resume_mode — determines whether the app should resume intro, guest analyzer, or signed-in account mode
  • pw_auth_notice — temporary auth notice state for reload-based transitions

Merge strategy

  1. User works as guest.
  2. Progress is saved locally.
  3. User signs in using Google or email.
  4. Application checks for guest data.
  5. If guest data exists, it is saved to the user’s Supabase row.
  6. Guest local storage is cleared afterward.

This design keeps onboarding friction low while still enabling account-based persistence later.

14. Readiness Scoring Engine

The scoring model in Pathwise is a weighted rule-based engine. It does not guess readiness with a black box. Instead, it transparently compares the user’s declared skills to role requirements.

Inputs

  • Selected role
  • Required skills for that role
  • Weight for each required skill
  • Current user skills
  • User proficiency level per skill

Skill level multipliers

LevelMultiplierReasoning
Beginner0.30Counts as exposure but not role-ready mastery
Intermediate0.65Strong partial readiness
Expert1.00Full skill contribution

Tier handling

  • Critical skills carry the highest importance.
  • Supporting skills matter but are not weighted as high as critical gaps in decision impact.
  • Optional skills add value but should not artificially inflate readiness.

Outputs

  • Total readiness score
  • Matched skills list
  • Missing skills list
  • Category breakdown percentages
  • Priority order for improvement

17. Security, Privacy, and Access Control

Security measures implemented

  • Authentication managed by Supabase instead of custom password handling
  • Frontend uses Supabase anon key only
  • Per-user row protection using Row Level Security policies
  • No service_role key exposed in the browser
  • Guest progress restricted to local browser storage
  • Signed-in persistence tied to user_id

Security limitations of the current MVP

  • The app depends on Supabase configuration correctness.
  • Guest data in localStorage is convenient but not sensitive-data hardened.
  • Email flows depend on provider reliability and template configuration.
  • There is no custom server-side validation layer because the architecture is serverless-client-centric.

Risk posture

For a student or hackathon-level deployment, the current security posture is strong enough because access control is delegated to a well-supported backend platform and data scope is limited. For large-scale production, additional audit logging, server-side abstractions, and centralized configuration management would be recommended.

20. Testing Strategy

The testing approach for Pathwise is focused on practical product QA rather than only isolated code-level assertions. Since the application is a flow-based interface, the most important validation areas are navigation, auth, persistence, calculation correctness, and state restoration.

Testing categories

  • Functional testing
  • UI/UX testing
  • Authentication testing
  • Persistence testing
  • Negative testing
  • Cross-state testing
  • Security testing
  • Regression testing

Primary QA focus areas

  • Does every authentication route actually lead into the analyzer?
  • Does reload preserve the correct page and data?
  • Do guest and signed-in flows remain consistent?
  • Do role cards, scope, and required skills stay synchronized?
  • Are results and action-plan sections built from the current role and skills only?

22. Quality Assurance Findings

During iterative QA, the most fragile parts of the application were not the scoring rules or the UI rendering, but the transitions around authentication, profile navigation, reload restoration, and email-based auth. Several bugs in these areas are typical for session-driven browser applications and were addressed during refinement.

Examples of areas that required hardening

  • Google sign-in returning to intro instead of opening the analyzer
  • Guest profile page not immediately reflecting current in-memory state
  • Analysis action redirecting back to setup after the loading animation
  • Header and dropdown visibility leaking onto screens where they were not expected
  • Animation overlap causing unwanted blinking in setup and action sections

The current build is significantly more stable because transitions now rely on explicit resume-state handling rather than only implicit UI state.

23. Limitations

  • The readiness score is rule-based and curated; it is not validated against hiring-market data at scale.
  • Email provider reliability depends on external configuration and template behavior.
  • Guest data remains browser-local and device-specific.
  • The role library is curated rather than exhaustive.
  • The application does not currently support resume upload or automated skill extraction.
  • The analyzer assumes self-reported skill accuracy from users.

25. Conclusion

Pathwise is more than a static career information page. It is a guided readiness platform that takes a user from uncertainty to action. Its strongest value lies in the combination of practical role selection, weighted skill analysis, personalized scoring, and next-step planning.

Technically, the project demonstrates frontend engineering, browser-state management, auth integration, cloud persistence, interface design, recommendation logic, and structured testing. Functionally, it focuses on a real student problem and provides output that is immediately useful.

For academic submission and project presentation, Pathwise represents a strong example of building a problem-focused product with clear workflows, real persistence, and a polished user experience.

Appendix A: SQL Schema and Policies

The following schema reflects the intended Supabase persistence model used by Pathwise.

create table if not exists public.user_progress ( id bigint generated always as identity primary key, user_id uuid not null references auth.users (id) on delete cascade, selected_role text not null, skills jsonb not null default '[]'::jsonb, score integer, created_at timestamptz not null default now() ); create index if not exists user_progress_user_id_created_at_idx on public.user_progress (user_id, created_at desc); alter table public.user_progress enable row level security; create policy "user_progress_select_own" on public.user_progress for select to authenticated using (auth.uid() = user_id); create policy "user_progress_insert_own" on public.user_progress for insert to authenticated with check (auth.uid() = user_id); create policy "user_progress_update_own" on public.user_progress for update to authenticated using (auth.uid() = user_id) with check (auth.uid() = user_id); create policy "user_progress_delete_own" on public.user_progress for delete to authenticated using (auth.uid() = user_id);
